## Section-by-Section Breakdown

### Section 1: Portfolio Health (from `trend_service.portfolio_snapshot()`)

| Data Point | Source | Why Executives Care |
|-----------|--------|-------------------|
| Customer count by risk level | `health_scores` table (latest per customer) | Instant read on portfolio stability |
| Avg health score + week-over-week change | `trend_service.health_trends()` | Trending up or down? |
| Upcoming renewals with ARR | `customers.renewal_date` + `deals.amount` | Revenue at risk |

**Rule:** Only show if there's a change from last brief. If portfolio is stable, compress to one line: "Portfolio stable — 32 customers, avg health 72/100, no change."

---

### Section 2: Attention Required (from alerts + health + tickets)

This is the **most important section** — executives scan this first.

| Data Point | Source | Trigger to Include |
|-----------|--------|-------------------|
| Customers with health decline >10 pts | `health_scores` (7-day delta) | Always include if exists |
| Open P0/P1 tickets stale >3 days | `tickets` table | Always include |
| Negative sentiment streaks (3+ calls) | `call_insights` | Always include |
| Renewals <60 days with bad health | `customers` + `health_scores` | Always include |
| Trending feature requests (3+ customers) | `tickets` (when Rule 1 from escalation rules is active) | Include if triggered |
| Unresolved escalations >3 days | `tickets` | Include if triggered |

**Rules:**
- Max 5 items (prioritize by severity: critical > high > medium)
- Each item: Customer name + what's wrong + suggested action (one line)
- If no items: "No critical attention items this period." (This is good news — show it.)

---

### Section 3: Ticket Summary (from `trend_service.ticket_velocity()`)

| Data Point | Source | Why Executives Care |
|-----------|--------|-------------------|
| Created / Resolved / Net Open (7 days) | `tickets` table aggregation | Are we keeping up with incoming volume? |
| P0/P1 aging count | `tickets` (severity + created_at) | Anything critical slipping? |
| Top issue pattern | `tickets` grouped by category/summary | Systemic problem detection |

**Rule:** If net open is negative (resolving more than creating), highlight positively. If P0/P1 aging > 0, always show.

---

### Section 4: Call Intelligence (from `call_insights` + `action_items`)

| Data Point | Source | Why Executives Care |
|-----------|--------|-------------------|
| Call count (7 days) | `call_insights.processed_at` | Team engagement level |
| Avg sentiment + negative call list | `call_insights.sentiment` + `sentiment_score` | Customer happiness trend |
| Pending action items (overdue) | `action_items` table or `call_insights.action_items` | Are we following through? |
| Key risk signals | `call_insights.risks` (aggregated) | Emerging problems from conversations |

**Rule:** Only list negative/mixed calls by customer name (executives don't need call titles). Risk signals: pick top 2 most mentioned across all calls.

---

### Section 5: Pipeline Snapshot (from `presales_funnel_agent` data + `deals` table)

| Data Point | Source | Why Executives Care |
|-----------|--------|-------------------|
| Open deals count + total value | `deals` table (stage != closedwon/closedlost) | Revenue pipeline health |
| Win rate (30 days) | `deals` (closedwon vs closedlost) | Are we closing? |
| Stalled deals count | `deals` (>30 days no update) | Deals going cold |
| Top deal (highest value open) | `deals` (sorted by amount) | Biggest opportunity awareness |
| Stage conversion rates | `presales_funnel_agent._compute_conversion_rates()` | Funnel efficiency |

**Rule:** If no deal stage changes this week, compress to one line: "Pipeline stable — 28 deals open ($4.8M), no stage changes."

---

### Section 6: Top Actions (AI-generated prioritization)

This section is **generated by Claude** based on all the data above. The system will:
1. Collect all alerts, stale tickets, overdue action items, at-risk renewals
2. Ask Claude to prioritize the top 3 most urgent actions
3. Format as a numbered list with severity indicator

**Rule:** Always exactly 3 items. If nothing urgent, shift to proactive actions (e.g., "Prepare QBR for customer X — renewal in 50 days").

## What We DON'T Show (keep it focused)

| Excluded | Why |
|----------|-----|
| Individual ticket details | Too granular — that's what `#cs-ticket-triage` is for |
| Full call transcripts/summaries | Too long — that's what `#cs-call-intelligence` is for |
| Agent execution logs | Internal operational detail |
| Individual deal probability breakdowns | Too granular — show only top deal |
| Health score calculation factors | They don't need the formula, just the score |
| QBR drafts | Separate channel, separate review process |
