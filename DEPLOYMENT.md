# Deployment Guide — CS Control Plane

Deploy the backend on **Render**, frontend on **Vercel**, database on **Neon Postgres**.

---

## 1. Backend (Render)

1. Go to [Render Dashboard](https://dashboard.render.com) → **New** → **Blueprint**
2. Connect your Git repo — Render reads `render.yaml` automatically
3. Set these environment variables in the Render dashboard:

| Variable | Value |
|----------|-------|
| `DATABASE_URL` | `postgresql+asyncpg://USER:PASS@ep-xxx.neon.tech/neondb?sslmode=require` |
| `SYNC_DATABASE_URL` | `postgresql://USER:PASS@ep-xxx.neon.tech/neondb?sslmode=require` |
| `JWT_SECRET_KEY` | Auto-generated by Render (see `render.yaml`) |
| `ANTHROPIC_API_KEY` | `sk-ant-...` |
| `CHROMADB_MODE` | `ephemeral` |
| `FRONTEND_URL` | *(set after Vercel deploys — e.g. `https://your-app.vercel.app`)* |
| `DASHBOARD_BASE_URL` | *(same as FRONTEND_URL — used for Slack deep-links)* |

**Optional integrations** (set if needed):
| Variable | Value |
|----------|-------|
| `JIRA_API_URL` | `https://hivepro.atlassian.net` |
| `JIRA_EMAIL` | Your Jira email |
| `JIRA_API_TOKEN` | Atlassian API token |
| `SLACK_ENABLED` | `true` |
| `SLACK_BOT_TOKEN` | `xoxb-...` |
| `SLACK_SIGNING_SECRET` | App signing secret |

4. Deploy. The start command runs `alembic upgrade head` automatically (creates all tables).
5. Verify: `GET https://<render-url>/api/health` → `{"status": "healthy"}`

> **Note:** No Redis/Celery worker needed for basic usage. Tasks run synchronously (eager mode).

---

## 2. Frontend (Vercel)

1. Import the repo in [Vercel](https://vercel.com) → set **Root Directory** to `frontend/`
2. Framework preset: **Vite** (auto-detected)
3. Set environment variables:

| Variable | Value |
|----------|-------|
| `VITE_API_URL` | `https://<render-url>/api` |
| `VITE_WS_URL` | `wss://<render-url>/api/ws` |

4. Deploy. Note the Vercel URL.

---

## 3. Cross-Link Frontend ↔ Backend

Go back to Render and set:
- `FRONTEND_URL` = your Vercel URL (enables strict CORS)
- `DASHBOARD_BASE_URL` = same Vercel URL (for Slack card deep-links)

Redeploy backend or wait for next deploy.

## 6. Slack Integration

Slack needs to reach your backend for two things: **interactive buttons** (Approve/Edit/Dismiss on agent cards) and **chat messages** (users typing queries in Slack channels).

**Local dev (current setup):** ngrok tunnel → `https://xxxx.ngrok-free.app` → localhost:8000

**Deployed on Render:** Render gives you a public URL, so ngrok is no longer needed. Update your Slack app config:

1. Go to [api.slack.com/apps](https://api.slack.com/apps) → select your app
2. **Interactivity & Shortcuts** → set Request URL to:
   ```
   https://<render-url>/api/webhooks/slack/interactions
   ```
3. **Event Subscriptions** → set Request URL to:
   ```
   https://<render-url>/api/webhooks/slack
   ```
   Slack will send a `url_verification` challenge — the endpoint handles it automatically.
4. Set these env vars on Render:

| Variable | Value |
|----------|-------|
| `SLACK_ENABLED` | `true` |
| `SLACK_BOT_TOKEN` | `xoxb-...` (from OAuth & Permissions) |
| `SLACK_SIGNING_SECRET` | (from Basic Information → App Credentials) |
| `SLACK_BOT_USER_ID` | Bot's user ID (find via `auth.test` API or Slack admin) |
| `SLACK_CHAT_CHANNEL` | Channel ID for chat queries (optional) |
| `SLACK_CH_TICKET_TRIAGE` | `#cs-ticket-triage` |
| `SLACK_CH_CALL_INTEL` | `#cs-call-intelligence` |
| `SLACK_CH_HEALTH_ALERTS` | `#cs-health-alerts` |
| `SLACK_CH_ESCALATIONS` | `#cs-escalations` |
| `SLACK_CH_DELIVERY` | `#cs-delivery` |
| `SLACK_CH_QBR_DRAFTS` | `#cs-qbr-drafts` |
| `SLACK_CH_PRESALES` | `#cs-presales-funnel` |
| `SLACK_CH_EXEC_DIGEST` | `#cs-executive-digest` |
| `SLACK_CH_EXEC_URGENT` | `#cs-executive-urgent` |

5. **Required bot scopes** (OAuth & Permissions → Scopes):
   `chat:write`, `channels:history`, `groups:history`, `im:history`, `mpim:history`, `users:read`

6. **Event subscriptions** (subscribe to these bot events):
   `message.channels`, `message.groups`, `message.im`, `app_mention`

> **Key difference from local dev:** No ngrok needed — Render's public URL replaces it. Just swap the webhook URLs in the Slack app config.

## 7. Data Sync Schedule

Both Jira and Fathom sync automatically via APScheduler (cron-style, fixed times).

| Source | Schedule (IST) | Behavior |
|--------|---------------|----------|
| **Jira** | Daily at **8:00 AM** + on startup | First run: 6-month catchup. Then incremental (last 25 hours). |
| **Fathom** | Twice daily at **6:00 AM** & **6:00 PM** + on startup | Syncs last 7 days each run. Deduplicates automatically. |

Timezone is configurable via `SYNC_TIMEZONE` (default: `Asia/Kolkata`).

Manual sync is also available via the API:
- `POST /api/jira/sync` — trigger Jira sync on demand
- `POST /api/fathom/sync?days=14` — trigger Fathom sync on demand

## 8. Known Limitations

- **Render free tier**: ~30s cold start after inactivity. First request will be slow.
- **Neon free tier**: ~6s cold start for first DB connection (pool pre-warming helps on subsequent requests).
- **WebSockets**: May not work on Render free tier depending on plan. The app degrades gracefully without them.
- **ChromaDB ephemeral**: RAG/vector search data is lost on restart. The "similar tickets" feature requires persistent ChromaDB.
- **No Redis**: Without Redis, all Celery tasks run synchronously. Fine for testing, not ideal for production load.
